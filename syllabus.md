# KimiCode 课件 · Vibe Coding 从入门到上线

> 面向零基础（非技术背景）大学生的纯线上九节实战课程
> 核心理念：**先描述想法，让 AI 帮你实现，不必先学语法**
> 每节课都做出一个能展示、能上手的真实成品 | 学员异步学习 + 在线讨论 + test code 验收

---

## 课程总览

| 节次 | 主题 | 当节产出 | 新学到的能力 | 需课前准备 |
|------|------|----------|--------------|------------|
| 第一节 | 先做一个很酷的东西 | 3D 可交互月球网页 | 拆解并复用一个有效 prompt | — |
| 第二节 | 造一个简易版 Manus | 会自己规划、调用多个 skill 的 agent | agent 概念、skill、orchestration prompt、MCP | IDE + CLI 配置 |
| 第三节 | 把 Manus 做成真正的项目 | 接真实 API 的 Next.js 网页（本地） | Git、Node、Next、.env、调 API | GitHub 账号 |
| 第四节 | 定时舆情分析网页 | 能登录、定时抓取、分析的看板 | 官方 API、数据库、登录验证、定时任务 | Reddit/GitHub API key |
| 第五节 | 做一个 Chrome 扩展 | 装进自己浏览器的实用小工具 | 扩展结构、content script、popup | — |
| 第六节 | 做一个微信小程序 | 能跑通的小程序 | 小程序结构、微信开发者工具、审核流程 | 小程序账号（实名慢，务必提前） |
| 第七节 | Xcode 的秘密 | 自己写的 iOS app | iOS 项目结构、SwiftUI、API 调用、应用签名 | Mac 电脑、Xcode、Apple ID |

**贯穿全程的一条原则**：命令记不住没关系，遇到报错就截图丢给 AI，让它教你修。这本身就是 vibe coding 的工作方式。

## 第一节课：先做一个很酷的东西（The Wow Moment）

**目标**：第一节课就让学生用一句话造出一个看起来很厉害的 3D 交互月球，打掉「编程很难」的心理门槛。

**时长**：约 60 分钟

### 流程

**1. 开场惊叹视频（5 min）**
观看演示视频：展示做好的月球——可以缩放、旋转看背面、背景有星星。视频中讲述「从 prompt 到成品」的全过程。学员在讨论区留言「猜测」制作时间，讲师后续揭晓：一句话。

**2. 认识工具（5 min）**
打开 KimiCode，介绍最核心的两个区域：
- **对话框**——你说话的地方
- **预览窗口**——结果实时出现的地方

不讲任何术语。

**3. 跟着做（15 min，异步自主）**
学员按照提供的教程文档，把这条已验证有效的 prompt 复制进去运行：

```
Create a HTML webpage based on the following cues. The page should be
a 3D interactive world titled Moon. It features a Moon in space, with
realistic SVG craters and meshes. Users can zoom in and out, and rotate
the moon (to see the dark side of it). The space should have little stars.
```

学员亲手在自己的环境里生成月球、亲手转一转，并在讨论区分享截图。

**4. 拆解这条 prompt（10 min）——本节最重要的知识点**

逐句讲为什么它有效，提炼出一个可复用的结构：

| 要素 | 作用 | 本例 |
|------|------|------|
| 是什么 | 标题 / 类型 | a 3D interactive world titled Moon |
| 主体 | 核心对象 | a Moon in space |
| 细节 | 让它真实 | realistic SVG craters and meshes |
| 交互 | 用户能做什么 | zoom in/out, rotate to see the dark side |
| 氛围 | 锦上添花 | little stars |

**结论**：描述得越具体，AI 给的结果越接近你想要的。

**5. 自己改（15 min）**
让学生在这条 prompt 上动手改，做出自己的版本。引导方向：
- 换成火星、土星（带光环）
- 改月球颜色、加一行说明文字
- 让星星会闪烁
- 加一个能点击的标记点

鼓励大胆试错，不满意就继续对 AI 说「再改一下……」。

**6. 成果分享（异步）**
学员在共享社区（如企业微信管理的群聊）展示各自的月球作品（截图或录屏），讲师和同学给反馈。总结心法：**把想法描述清楚，就是编程。**

### 课后作业
用今天学的 prompt 结构，做一个属于自己的 3D 场景（地球、太阳系、虚构星球），或给月球加一个新功能。

## 第二节课：造一个你自己的简易版 Manus

**目标**：从「我会让 AI 做一件事」跨越到「我造了一个能自己规划、调动多种能力、把复杂任务自动跑完的 agent」。

**核心立意**：Manus 这类 agent 的厉害之处，不是会某一个绝活，而是**会规划 + 会调用一堆不同的 skill**把一件复杂的事从头做到尾。
> **agent = 会规划的大脑 + 一堆 skill**

**时长**：约 75 分钟

### 流程

**1. 开场演示视频（5 min）**
观看讲师录制的演示视频：展示简易版 Manus 的完整运行过程。总任务示例：「帮我研究「电动车 vs 燃油车」，产出一份报告和一套幻灯片」。视频展示它自己拆步骤、逐个调用不同 skill、最后吐出成品的全过程。
> 「上节课我们一句话造了个月球，这节课我们造一个会替你打工的助理。"
> 「上节课我们一句话造了个月球，这节课我们造一个会替你打工的助理。」

**2. 什么是 agent（10 min）**
只用一个比喻，不堆术语：
- 上节课的 AI 像一个「一次性工人」——你说一句，它做一件事
- agent 像一个「项目负责人」——你给它大目标，它自己想「分几步、每步用什么本事」，逐步完成，还能根据结果调整
- 它的「本事」就是上节课的 **skill**；agent 强不强，取决于手里有多少趁手的 skill

**3. 环境准备（课前 + 课程初期）**
学员按照详细的配置教程，装好 IDE（VS Code）、装好并登录 KimiCode CLI、跑通第一条命令。
> ⁉️ **提供完整的课前预习视频/手册。学员如遇问题可在讨论区提问，讲师或助教在 24 小时内回复。不阻桶后续学习。**

**4. 动手造（核心环节，约 40 min）**

**第一步——定一个总任务（5 min）**
选一个「一步做不完、必须分好几步」的目标：
- 研究型：研究某主题 → 报告 + 幻灯片
- 策划型：规划三天旅行 → 行程表 + 预算表 + 打包清单
- 产品型：一句创意 → 落地页 + 宣传文案

任务要「够大」，大到必须拆步骤、用到不止一种能力。

**第二步——准备这个 agent 的「本事」（多个 skill）（15 min）**
以研究型为例，至少需要：

| skill | 负责什么 |
|-------|----------|
| 调研 skill | 怎么找资料、判断可信、整理要点 |
| 写报告 skill | 结构、语气、格式规范 |
| 做幻灯片 skill | 每页怎么排、风格统一 |
| 统一风格 skill | 颜色、字体、品牌调性 |

讲师通过视频演示现成 skill（如 superpower 系列）的**开 vs 不开**效果对比，让学员直观感受 skill 的价值。然后每位学员**至少自己写一个 skill**（最简单：「我的风格偏好 skill」或「我的报告结构 skill」），在提交作业环节提交代码和运行结果截图，讲师反馈。

**第三步——写「大脑」：让它学会自己规划和调度（15 min）**
这是和第一节最不同的一步。带学生写 orchestration prompt（agent 主指令），教 AI 三件事：
1. **先规划**：拿到总任务，先列步骤，不急着动手
2. **再按需调用**：每一步该用哪个 skill 就调哪个
3. **串起来**：上一步产出当作下一步输入，最后汇总成完整交付物

可填空骨架模板：

```
你是一个项目负责人。收到任务后，先把它拆解成步骤，并说明每步用哪个能力；
然后逐步执行，每一步调用对应的 skill；
全部执行完后，汇总所有产出成为完整的交付物。

任务是：______
```

让学生填进自己的总任务跑起来，看它自己「先想后做」——这就是本节的高光时刻。

**第四步（进阶/选做）——接 MCP，让它够得到真实世界**
MCP 让 agent 从「凭脑子工作」升级到能读真实文件、查真实数据、把产出存到网盘/日历。讲师通过视频演示接一个最直观的 MCP（文件系统或网盘）的完整过程，展示 agent 把报告和幻灯片真正存成文件。学员参照教程在自己的环境中实践。

**5. 自己扩展（10 min）**
给自己的 agent 再加一项本事：多写一个 skill、多接一个 MCP，或给总任务加一个新步骤（如「自动写一封推荐这份报告的邮件」）。

**6. 收尾（5 min）**
> 「上节课你学会把一个想法说清楚；这节课你学会把一个复杂目标交给一个会规划、会调动多种本事的 agent。你今天造的，就是一个属于你自己的迷你版 Manus。」

### 学习成果提交
学员需在规定时间内提交：
1. 给简易 Manus 至少配齐 **3 个 skill** 的代码
2. 完成一个完整的多步任务的运行结果（录屏或截图）
3. 一份「运行记录」：它把任务拆成了几步、每步用了什么 skill、踩了什么坑、怎么靠 AI 修好的

讲师或助教在 3 个工作日内给反馈。

### 学习节奏
信息量大。学员按自己的进度学习，讲师建议：IDE/CLI 配置全部课前完成；「自己写 skill」先写一个、其余可选；重点控制「规划 + 调度」逻辑。遇到卡点在讨论区提问，讲师 24 小时内回复。若某人进度慢，可在第三节继续完成 MCP 部分。

## 第三节课：把简易 Manus 做成一个真正的项目

**目标**：把 agent 从 KimiCode 环境「搬」成一个真正的 Next.js 网页项目，第一次接触工程化四件套——版本管理、运行时、框架、密钥管理，并接真实 API。**本节只在本地跑通，不部署**（部署留到第七节）。

**时长**：约 75 分钟

### 流程

**1. 为什么要「搬家」（5 min）**
上节课的 agent 困在对话里——别人用不了、改了容易丢、接不了你自己的密钥。做成真正的网页项目，这些都解锁。

**2. Git：给项目上「存档点」（15 min）**
用游戏存档类比：commit = 存档，出错能读档回去，多人协作还能合并进度。只教核心几步：
- 初始化
- add
- commit
- 看历史
- 回退

建 GitHub 账号，把项目推上去。

**3. Node.js + Next.js：认识工作台（15 min）**
用最朴素的话讲清：
- **Node** = 让 JavaScript 在你电脑上跑起来的引擎
- **Next** = 帮你快速搭网页的框架，前端页面和后端逻辑都能写

学员跟随教程，让 AI 帮忙用 Next 起一个最小项目，跑起来看到本地页面，体验「改文件、网页自动刷新」。在讨论区分享截图/录屏验证完成。

**4. 环境变量 .env：藏好你的钥匙（10 min）——本节最重要的工程意识**
- API key 是你的钥匙，**绝对不能直接写在代码里、更不能推到 GitHub 上**（别人捡到能拿你的账号花钱）
- 把密钥放进 `.env` 文件
- 把 `.env` 加进 `.gitignore`，让它永远不会被上传

> 🔑 这个习惯越早建立越好。

**5. 接真实 API，让 agent 长出「真本事」（25 min）——核心动手环节**
让简易 Manus 接上真实大模型 API（用 `.env` 里的 key），从「在对话框里跑」变成「在自己的网页里、用自己的密钥跑」。做出一个能用的界面：
- 一个输入框接收总任务
- 后端调用 API 做规划和分步执行
- 结果展示在页面上

能力强的把第二节的多 skill 调度逻辑也搬进来。

**6. 收尾（5 min）**
> 「今天你的 agent 从一个聊天记录，变成了一个真正属于你自己的网页应用——有版本、有引擎、有框架，钥匙也藏好了。」

### 学习成果提交
学员需在规定时间内提交：
- 项目 GitHub 链接（含完整 commit 历史）
- 在本地跑通接了真实 API 的简易 Manus 的运行截图
- 一份清单：确认 `.env` 没有被上传（安全红线检查）

讲师在 3 个工作日内验收并反馈。

## 第四节课：做一个会自己定时跑的舆情分析网页

**目标**：从「用户点一下才动」进阶到「自己定时抓数据、存起来、做分析」。第一次接触数据获取、用户系统、数据持久化、定时任务。建议主攻 **Reddit 或 GitHub**——都有官方 API，合规、稳定、好上手。

**时长**：约 80 分钟

### 流程

**1. 看成品演示视频（5 min）**
观看讲师录制的演示视频：展示一个完整的舆情面板运行流程——定时抓取某话题在 Reddit/GitHub 上的讨论，自动分析情绪和热点，登录后才能看到自己的看板。

**2. 数据从哪来：优先用官方 API，而不是硬爬（15 min）**
讲师讲解并通过视频演示正确观念：**能用官方 API 就别硬爬。**
- Reddit / GitHub 都提供官方 API：合规、稳定、附带结构化数据
- 教学员申请 API key（存进 `.env`），用 API 拉一个话题的帖子/issue 回来
- 尊重使用条款和频率限制，别把人家服务器打挂——这是技术规范也是基本礼貌
- 确实只能从网页拿的数据，再简单演示基础抓取，同样守规矩

**3. 数据库：把抓回来的东西存下来（15 min）**
为什么要数据库——抓回来的数据不能只放内存里，关掉就没了。用上手简单的方案（如 Supabase 这类带现成后端的）：
- 建一张表
- 把抓到的内容存进去
- 再读出来展示

体会「写入—读取」这个最基本的循环。

**4. 登录验证：这是谁的看板（15 min）**
为什么要登录——不同用户看不同数据，有些内容不该公开。用现成认证方案（Supabase Auth / Clerk 这类）加注册登录，做到「登录后才能看自己的看板」。先会用、理解它解决了什么问题即可。

**5. 分析 + 定时跑（15 min）**
- 用大模型 API 对抓回来的讨论做情绪/热点分析（复用第三节经验）
- 讲「定时任务」：让程序每隔一段时间自己跑一次抓取和分析
- 本地先用最简单的定时方式演示效果，**真正的云端定时留到第七节落地**

**6. 收尾（5 min）**
> 「今天你的应用第一次有了记忆（数据库）、有了门禁（登录）、还会自己干活（定时）。这已经是一个有模有样的真实产品了。」

### 学习成果提交
学员需在规定时间内提交：
1. 完整的舆情面板项目（GitHub 链接）
2. 运行演示录屏（展示登录、抓取、分析全流程）
3. 一份技术说明文档：数据来自哪个官方 API、如何避免触犯频率限制

讲师在 3 个工作日内验收。

## 第五节课：做一个 Chrome 扩展

**目标**：换一种产品形态——不是一个网页，而是一个「长在别人网页上」的小工具。理解扩展怎么运行，做出一个能装进自己浏览器、立刻能用的东西。

**时长**：约 70 分钟

### 流程

**1. 看成品演示视频（5 min）**
观看讲师录制的演示视频，展示一两个实用小扩展的运行效果：
- 一键总结当前网页
- 把选中文字翻译/改写
- 把当前页面信息存进第四节的数据库

**2. 扩展是怎么回事（15 min）**
用最直白的方式讲清结构：

| 组成 | 作用 |
|------|------|
| manifest | 配置文件，说明这个扩展是谁、要什么权限 |
| content script | 运行在你正在浏览的网页上的代码 |
| popup | 工具栏点开的弹出小窗（可选） |

讲清「哪段代码在哪里跑」就够了。

**3. 跑通第一个扩展（20 min，异步自主）**
学员跟随教程，让 AI 帮忙生成一个最小扩展，在自己的 Chrome 里用「开发者模式」加载，看到它出现在工具栏、点一下有反应。在讨论区分享截图。
> 「我做的东西装进了我自己的浏览器」——这个时刻很有冲击力。

**4. 做一个真正有用的（25 min）——核心动手环节**
做一个调用大模型 API 的小扩展，比如总结或改写当前网页选中的内容（复用前几节接 API 经验）。能力强的让它和第四节项目联动——把网页内容一键存进自己的数据库。

**5. 收尾（5 min）**
> 「同一身本事（接 API、连数据），可以做成网页，也可以做成扩展。产品形态不同，内核是相通的。」

### 学习成果提交
学员需在规定时间内提交：
1. 完整的扩展项目代码（GitHub 链接或提交）
2. 扩展的使用演示录屏（展示装进浏览器 + 完整功能）

讲师在 3 个工作日内验收。

## 第六节课：做一个微信小程序

**目标**：再换一种形态——从浏览器走到手机、走到微信生态。面向中国大学生特别实用，小程序是国内分发产品最贴近用户的方式之一。

**时长**：约 75 分钟

### 流程

**1. 看成品演示视频（5 min）**
观看讲师录制的演示视频：展示一个简单但完整的小程序运行效果，最好和前面项目呼应（如把舆情面板或某个工具做成小程序版）。

**2. 小程序是怎么回事（15 min）**
讲清它和普通网页的区别：
- 跑在微信里、用微信的账号体系
- 有自己的一套页面写法和组件
- 发布前要走微信审核

**3. 环境与账号（课前 + 课程初期）**
- 学员提前按教程装好微信开发者工具

## 第九节课：AI 的自我进化——一个 Loop，打造自我改写的 Agent

**目标**：突破传统的「静态 prompt」思维，理解 AI 系统如何通过反馈和评估不断自我优化。学员将基于 OpenClaw 框架构建一个能够自我改进的 agent——它会在执行任务过程中，根据结果的好坏自动调整自己的思路、改写自己的 prompt，形成一个自洽的改进循环。这是 AI 工程中最前沿的话题：**从静态工具到自进化系统**。

**时长**：约 75 分钟

### 学习目标

完成本节课后，学员应能：
1. 理解「自我改进循环」的核心逻辑（执行 → 评估 → 反思 → 改写 → 再执行）
2. 认识 OpenClaw 框架的 5 大核心模块（Goal、Executor、Evaluator、Revisor、Loop）
3. 实现一个「会自己调整策略」的多步骤 agent
4. 为 agent 设计合理的「评估标准」（什么叫「做得好」）
5. 调试一个自我进化系统，观察 prompt 如何被自动改写

### 流程

**1. 一个故事：从静态到活的（开场演示视频 5 min）**

讲师演示：
- **场景 A（坏的方式）**：一个固定 prompt 的 agent，给出一个任务，它生成一个方案，可能很差，就此打住。
- **场景 B（好的方式）**：同一个 agent，第一次可能也很差，但它自己评价「这个方案不够好」，然后自动改进自己的思路，第二次更好，第三次更好……最后得到一个令人满意的答案。

讲述核心观念："静态 prompt 是说明书，自我进化是生命"。引出 OpenClaw 框架。

**2. 核心概念：五环节自我改进循环（讲解视频 10 min）**

讲师通过图示和代码演示，讲清楚这 5 个环节：

- **Goal（目标）**：定义 agent 要达成什么。例如「写一个产品介绍文案，要求文字优美、转化率高、没有错别字」。

- **Executor（执行器）**：真正干活的部分。给定一个 prompt 和输入，它生成一个初版方案。

  ```python
  # 伪代码
  def executor(goal, previous_prompt, task_input):
      # 用当前的 prompt 去执行任务
      response = llm(prompt=previous_prompt, input=task_input)
      return response
  ```

- **Evaluator（评估器）**：打分官。看看这个方案有没有达成目标，给出具体的反馈（不光是「不好」，而是「差在哪里」）。

  ```python
  def evaluator(goal, output):
      # 对着目标，评价输出质量
      score = check_clarity(output)  # 清晰度
      score += check_grammar(output)  # 语法
      score += check_alignment_with_goal(output, goal)  # 与目标的符合度
      return score, detailed_feedback
  ```

- **Revisor（修改器）**：反思官。拿到评估反馈，生成一个改进版本的 prompt，让 executor 下一轮能做得更好。这是「自我改写」的核心。

  ```python
  def revisor(current_prompt, evaluator_feedback, goal):
      # 根据反馈，改进 prompt
      new_prompt = llm(
          instruction=f"You are a prompt engineer. The current prompt is: {current_prompt}. "
                       f"The evaluator feedback is: {evaluator_feedback}. "
                       f"Goal is: {goal}. Rewrite the prompt to be better.",
      )
      return new_prompt
  ```

- **Loop（循环）**：把上面 4 个环节串联起来，重复执行，直到评估分数收敛或达到目标。

  ```python
  def evolve_loop(goal, task_input, max_iterations=5):
      current_prompt = initial_prompt  # 从某个初始 prompt 开始
      best_output = None
      best_score = 0
      
      for i in range(max_iterations):
          # 第一步：执行
          output = executor(goal, current_prompt, task_input)
          
          # 第二步：评估
          score, feedback = evaluator(goal, output)
          
          # 记录最好的结果
          if score > best_score:
              best_score = score
              best_output = output
          
          # 第三步：修改（如果还没有完全达标，就继续改）
          if score < TARGET_SCORE:
              current_prompt = revisor(current_prompt, feedback, goal)
          else:
              break
      
      return best_output, current_prompt
  ```

讲解重点：**这个循环不是「人工调试」，而是 AI 自己在调自己的 prompt，这就是自进化。**

**3. 跟着做：实现一个自我进化的「文案生成器」（30 min，异步自主）**

学员基于教程文档，用提供的 OpenClaw 框架（或兼容实现）完成以下任务：

**任务 1：定义一个简单的目标和评估器**
```python
goal = "生成一个吸引力强、无错别字、适合 25-35 岁女性阅读的护肤品文案"

def my_evaluator(output):
    # 检查包含关键词
    keyword_score = 1 if "成分" in output and "效果" in output else 0
    
    # 检查没有错别字（这里用 LLM 辅助检查）
    grammar_check = llm(f"Check if this has typos: {output}")
    grammar_score = 1 if "no typo" in grammar_check else 0
    
    # 长度合理（200-300 字）
    length_score = 1 if 200 <= len(output) <= 300 else 0.5
    
    return keyword_score + grammar_score + length_score
```

**任务 2：初始化 executor 和 revisor**
```python
# executor：用一个初始 prompt 调用 LLM
def my_executor(current_prompt, product_info):
    return llm(prompt=current_prompt, input=product_info)

# revisor：根据反馈改进 prompt
def my_revisor(current_prompt, feedback, goal):
    instruction = f"""
    你是提示词工程师。
    当前提示词：{current_prompt}
    反馈：{feedback}
    目标：{goal}
    
    根据反馈，改进提示词，使下一轮生成的文案更好。
    直接输出改进后的提示词。
    """
    return llm(prompt=instruction)
```

**任务 3：运行自我进化循环**
```python
# 初始 prompt（故意设计得有点差，这样才能看到改进过程）
initial_prompt = "写一个护肤品文案。"

goal = "生成一个吸引力强、无错别字、适合 25-35 岁女性阅读的护肤品文案"
product_info = "产品：XX 精华液，主要成分：透明质酸、烟酰胺，功效：补水保湿、淡化细纹"

# 运行循环（教程提供的框架会处理大部分逻辑）
final_output, evolved_prompt, history = evolve_loop(
    goal=goal,
    executor=my_executor,
    evaluator=my_evaluator,
    revisor=my_revisor,
    initial_prompt=initial_prompt,
    task_input=product_info,
    max_iterations=5,
    target_score=2.5
)

print("最终文案：", final_output)
print("演进过程：", history)  # 可以看到每一轮的 prompt 是如何被改写的
```

**学员需要提交**：
- 运行结果的截屏（包括初始 prompt、每一轮的改进、最终得到的文案）
- 观察报告：写下你观察到的「prompt 是如何一步步被改进的」（不少于 200 字）

**4. 进阶讨论：自我进化的边界与陷阱（讲解视频 5 min）**

讲师讲解：
- **什么时候有效**：任务目标清晰、有好的评估标准时，自我进化效果最好。例如「写代码、优化营销文案、规划工作流」都很适合。
- **什么时候失效**：评估标准模糊或 evaluator 本身有偏见时，agent 会陷入「自我欺骗」（例如为了刷分数，生成看似有道理但实际荒谬的东西）。
- **计算成本**：每一轮都要调用 LLM，迭代次数多的话成本高。需要在质量和成本之间权衡。
